# Changelog

## [Unreleased]

### Added
- Heartbeat now reports `lan_ip` â€” this gateway's own primary RFC1918 LAN IPv4. The server uses it to rewrite a co-located service's loopback target (`127.0.0.1`) to the host's real LAN address when that route has failed over to a sibling gateway (HTTP `X-Gateway-Target` and L4/SSH/RDP config push). The address is picked from physical-LAN interfaces only (WireGuard/docker/VPN excluded), preferring the subnet that contains the host default route; public/loopback/link-local addresses are never reported. A host with no private LAN address reports `null`, and the server then degrades safely (HTTP 502 / no L4 listener) instead of mis-forwarding. Companion side of the server's gateway-loopback-failover feature.

### Fixed
- Security: cleared the high-severity `tmp` advisory (GHSA-52f5-9888-hmc6 / GHSA-ph9p-34f9-6g65 â€” symlink/path-traversal arbitrary write) by pinning `tmp â‰Ą 0.2.7` via `overrides`. `tmp` is reached only transitively through the dev-only `@stryker-mutator/core â†’ @inquirer/prompts â†’ external-editor` chain (mutation testing) and is never shipped in the production image (`npm ci --omit=dev`); the override avoids the breaking Stryker 9 major bump that `npm audit fix --force` proposed. Also pinned production `qs â‰Ą 6.15.2` (GHSA-q8mj-m7cp-5q26 â€” `qs.stringify` DoS, reached via `express`/`body-parser`). Lockfile + `overrides`-only change; no source change, all tests green. The 2 remaining `ajv` advisories are moderate, dev-only (Stryker), and below the CI `--audit-level=high` gate â€” left untouched to avoid forcing ESLint's `ajv@6` to a breaking major.
- A failed TCP listener bind no longer crashes the gateway. `TcpProxyManager.setRoutes` applied L4 routes in one sequential `await` loop, so the first `EADDRINUSE` (a host process already owns the port under `network_mode: host`, or two routes share a `listen_port`) rejected the whole method â€” and since `setRoutes` runs inside an `async` EventEmitter `'change'` listener in `bootstrap.js`, that rejection was unhandled. On Node's default `unhandledRejection` mode (`throw`) it terminated the process, and because a persistently-conflicting config is re-applied on every poll (and on the initial bootstrap poll), the result was a **crash-loop** on a remote, hard-to-reach gateway. Each route is now applied in isolation: a bind failure is logged (`routeId`, `listenPort`, `EADDRINUSE`) and skipped, sibling routes are unaffected, and `setRoutes` resolves. A process-level `unhandledRejection` handler in `src/index.js` is added as a final safety net so any future async-listener rejection logs loudly and keeps the gateway alive instead of crash-looping.
- WebSocket upgrades are now proxied through the gateway. `src/proxy/http.js` only ever called `proxy.web()` from the `request` handler and never registered an `'upgrade'` handler, so Node silently dropped WebSocket upgrades â€” breaking any backend that relies on them behind a gateway route, most visibly the **Proxmox noVNC/xterm.js console** (which tunnels VNC over a WebSocket on the same `:8006` backend port, *not* on a separate VNC port). Plain HTTP worked, the console hung. The proxy now forwards upgrades via `proxy.ws()`, reusing the same `X-Gateway-Target-Domain` routing and the per-route `backend_https` flag to pick `ws://` vs `wss://`. `X-Gateway-*` headers are stripped on the ws path too (`proxyReqWs`), and the shared `error` handler now distinguishes `ServerResponse` (web) from `Socket` (ws) so a failed upstream upgrade can't crash the gateway. No server-side, Caddy or route-config change is required â€” WebSocket support is transparent for every gateway route.
- Security: bumped `axios` 1.15.0 â†’ 1.16.1 and pinned `fast-uri` â‰Ą 3.1.2 (via `overrides`) to clear all high-severity `npm audit` advisories â€” prototype-pollution / NO_PROXY-bypass gadgets in axios and path-traversal in fast-uri. `fast-uri` is reached only transitively through the dev-only Strykerâ†’ajv chain and is never shipped in the production image (`npm ci --omit=dev`); axios is the production HTTP client for heartbeat + config sync. Lockfile-only change to dependencies, no source change.
- Heartbeat and config-sync now use dedicated HTTP/HTTPS agents with `keepAlive: false`. Node 20's global agent defaults to keep-alive, which caused stale TLS sockets to be reused after network hiccups â€” server responded with `TLS alert 80 (internal_error)` on every subsequent request until the container was restarted.
- Lint: switched to `plugin:security/recommended-legacy` for compatibility with ESLint 8 (`eslint-plugin-security@3` dropped legacy-config support from `recommended`).
- `wireguard.runCommand` accepts a `timeoutMs` option and `getStatus()` now uses `1500ms` so a wedged `wg show` (rare but observed after TUN-device hiccups) can no longer block the self-check or heartbeat collection indefinitely. `wg-quick up/down` continues to run unbounded since boot-time setup can legitimately take longer.
- `startHeartbeatTicker` caps `getHealth()` at 8s (`Promise.race`). When the cap fires the gateway still sends a heartbeat with `{ overall_healthy: false, reason: 'health_collection_timeout' }` so the server's state machine sees liveness and the operator sees the timeout reason in `last_health` for diagnosis. Default cap is below the heartbeat HTTP-timeout (10s) so the next tick can land before the previous overruns it.

### Changed
- Self-check wiring centralised in `src/health/selfCheckRunner.js`. Both `/api/status` and the heartbeat ticker now call a single `runHealthCheck()` closure built once at bootstrap time â€” previously the route-list, DNS resolver and reachability probe were duplicated across two call-sites in `bootstrap.js`. L4 routes get a synthesized `domain: 'l4:<port>'` label in `route_reachability` (was undefined for the heartbeat path before).
- Mutation-test coverage extended for `wol._computeBroadcast` (octet-range boundaries, /16, /32), `wol.validateMac` (length / hex), `config.isRfc1918` (exact 10/8, 172.16/12, 169.254/16 and 192.168/16 boundaries) and `proxy/router` (`wol_enabled` ternary). Aim is to retire the `break = 45` floor in stryker.conf.json on the next mutation run.
- Mutation `break` threshold lowered to 45 to reflect actual current score (49.58). `high`/`low` remain at 90/80 as aspirational targets â€” surviving mutants in `wol.js`, `config.js`, `router.js` need follow-up test coverage.
- Docker HEALTHCHECK now runs a dedicated `src/healthcheck.js` that parses `/config/gateway.env` directly. The previous inline `node -e` probe relied on `process.env.GC_TUNNEL_IP` / `GC_API_PORT`, but those variables live in the volume-mounted env file and are never exposed to the container's Docker ENV â€” the Node runtime reads the file and keeps the values in-memory, so a separate probe process sees nothing. Start-period raised to 60s to accommodate WG bring-up.
- `/api/health` is now registered before the `/api` auth-middleware mount. Previously the auth guard matched first and rejected every unauthenticated probe with 401, so the health endpoint was unreachable despite its comment claiming otherwise.
